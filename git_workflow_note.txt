MY STANDARD GIT WORKFLOW (NEW PROJECT)

1. MAIN BRANCH RULE
- main is always stable and production-ready
- never experiment directly in main

2. INITIAL SETUP
- create project
- initialize git
- push first version to main

3. CREATE DEV BRANCH
- create dev from main
  git checkout -b dev
  git push -u origin dev

- dev starts as exact copy of main

4. DAILY DEVELOPMENT
- work only in dev
- commit often with clear messages
  git add .
  git commit -m "Dev: feature / fix"

- push dev when needed (backup, end of day, big changes)
  git push

5. SWITCHING BRANCH SAFETY
- always check status before switching
  git status

- if unsure, stash changes
  git stash

6. MOVING CODE FROM DEV TO MAIN
- switch to main
  git checkout main

- copy only required files from dev
  git checkout dev -- path/to/file

- review changes
  git diff

- commit and push to main

7. MISTAKE RECOVERY
- uncommitted change:
  git restore file.txt

- committed mistake (safe way):
  git revert <commit-hash>

- never force reset pushed main

8. CORE RULE TO REMEMBER
Commit before switching branches.
Push when I want safety.
